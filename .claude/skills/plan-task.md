# Plan Task - タスク計画ガイド

このスキルは、新しいタスクのステアリングファイルを作成するために必要な情報を構造化してヒアリングします。

## ヒアリング項目

以下の項目について順番に質問し、回答を収集します：

### 1. タスクの基本情報
- **タスク名**: 簡潔な名前（例: add-tag-feature, fix-filter-bug）
- **概要**: 何を実装・修正するか（1-2文）
- **背景・目的**: なぜこのタスクが必要か

### 2. 影響範囲
- **対象サービス**: □ Frontend □ BFF □ Backend □ E2E
- **既存機能への影響**: 破壊的変更はあるか？
- **他のタスク・機能との依存関係**: あるか？

### 3. 技術的変更点

#### 3.1 API契約
- **API変更の有無**: あり / なし
- **変更内容**: 新規エンドポイント追加 / 既存エンドポイント修正 / 削除
- **後方互換性**: 維持される / 破壊的変更

#### 3.2 データベース
- **DB変更の有無**: あり / なし
- **変更内容**: テーブル追加 / カラム追加・変更 / インデックス追加
- **マイグレーション方針**: Flyway使用

#### 3.3 新規ユビキタス言語
- **新しいドメイン用語・ビジネス用語の有無**: あり / なし
- **用語名（日本語）**:
- **用語名（英語）**:
- **定義**:
- **コード上の命名規則**:
- **`docs/glossary.md`への追加**: 必要 / 不要

### 4. 非機能要件
- **パフォーマンス要件**: レスポンスタイム、スループット等
- **セキュリティ要件**: 認証・認可、データ保護等
- **J-SOX対応**: 監査ログ、承認フロー等
- **その他**: 可用性、拡張性等

### 5. 受け入れ条件
- **機能要件**: どうなれば完了とみなすか（箇条書き）
- **品質要件**: テストカバレッジ、パフォーマンス基準等

### 6. 制約事項
- **技術的制約**: 使用できない技術、ライブラリ、アプローチ
- **スケジュール制約**: 期限、マイルストーン
- **リソース制約**: 人員、予算、環境
- **ビジネス制約**: 法規制、社内ルール、既存契約
- **その他の制約**:

### 7. その他
- **参考資料**: 関連ドキュメント、Issue番号等
- **特記事項**: その他の重要な情報

---

## 実行フロー

1. **ヒアリング開始**: 上記の項目を順番に質問
2. **回答の整理**: 収集した情報を構造化
3. **ステアリングディレクトリ作成**: `.steering/[YYYYMMDD]-[タスク名]/`
4. **ファイル生成**:
   - `requirements.md`: タスクの要求内容
   - `design.md`: 実装設計
   - `tasklist.md`: Agent別タスクリスト
5. **永続的ドキュメント更新提案**: `docs/`の更新が必要な場合は提案
   - `docs/glossary.md`: 新規ユビキタス言語の追加
   - `docs/service-contracts.md`: API契約の更新
   - その他関連ドキュメント

---

## 使用方法

```
/plan-task
```

このコマンドを実行すると、対話形式でヒアリングが開始されます。

---

## 注意事項

- すべての項目に回答する必要はありません（該当しない項目は「なし」で構いません）
- 不明な点があれば、後で調査・確認することも可能です
- ヒアリング終了後、ステアリングファイルのレビューを行います
