# Review Implementation - 成果物レビュー

このスキルは、Agent Teamsで実装された成果物がステアリングファイルの仕様に準拠しているかを包括的にレビューします。

## Parameters

- `$1`: ステアリングディレクトリ名（例: `20250407-frontend-bff-only`）

## 使用方法

```
/review-implementation 20250407-frontend-bff-only
```

---

## レビュー項目

### 1. ステアリングファイル準拠性チェック

#### 1.1 requirements.md との整合性
- [ ] 受け入れ条件がすべて満たされているか
- [ ] ユーザーストーリーが実装されているか
- [ ] 制約事項が遵守されているか
- [ ] 影響範囲（対象サービス）が正しいか

#### 1.2 design.md との整合性
- [ ] 設計通りのアーキテクチャで実装されているか
- [ ] データ構造が設計通りか
- [ ] コンポーネント構成が設計通りか
- [ ] 実装アプローチが設計に従っているか

#### 1.3 tasklist.md との整合性
- [ ] すべてのタスクが完了しているか
- [ ] Agent別の担当範囲が正しく実装されているか
- [ ] タスク間の依存関係が解決されているか

### 2. 機能過不足チェック

#### 2.1 機能過剰チェック
- [ ] 仕様にない余計な機能を追加していないか
- [ ] スコープ外の実装がないか
- [ ] 過度な抽象化・汎用化をしていないか

#### 2.2 機能不足チェック
- [ ] 必須機能がすべて実装されているか
- [ ] エッジケースの処理が漏れていないか
- [ ] エラーハンドリングが適切か

### 3. API契約準拠性チェック

- [ ] `contracts/openapi/` の仕様と実装が一致しているか
- [ ] リクエスト・レスポンスの型が正しいか
- [ ] HTTPステータスコードが仕様通りか
- [ ] エラーレスポンスの形式が統一されているか
- [ ] バージョニングが適切か

### 4. 用語統一チェック

- [ ] `docs/glossary.md` のユビキタス言語が使われているか
- [ ] 変数名・関数名・型名が用語集に準拠しているか
- [ ] コメント・ログメッセージで用語が統一されているか
- [ ] 新規用語が追加された場合、glossary.mdが更新されているか

### 5. セキュリティチェック

#### 5.1 認証・認可
- [ ] `docs/security-guidelines.md` の認証方式に準拠しているか
- [ ] 認可チェックが適切に実装されているか（権限チェック）
- [ ] セッション管理が安全か（HttpOnly Cookie等）
- [ ] CSRF対策が実装されているか（該当する場合）

#### 5.2 入力検証
- [ ] すべてのユーザー入力がバリデーションされているか
- [ ] SQLインジェクション対策がされているか
- [ ] XSS対策がされているか（フロントエンド）
- [ ] パストラバーサル対策がされているか

#### 5.3 データ保護
- [ ] 機密情報（パスワード、トークン等）が適切に暗号化されているか
- [ ] パスワードがハッシュ化されているか（bcrypt等）
- [ ] 環境変数から機密情報を読み込んでいるか（ハードコード禁止）
- [ ] ログに機密情報が出力されていないか

#### 5.4 その他セキュリティ
- [ ] 依存ライブラリに既知の脆弱性がないか
- [ ] HTTPSが強制されているか（本番環境）
- [ ] セキュリティヘッダーが適切に設定されているか

### 6. パフォーマンスチェック

#### 6.1 データベースクエリ
- [ ] N+1問題が発生していないか
- [ ] 適切なインデックスが設定されているか
- [ ] 不要なカラムを取得していないか（SELECT *の乱用）
- [ ] ページネーションが実装されているか（大量データの場合）

#### 6.2 API・ネットワーク
- [ ] 不要なAPI呼び出しがないか
- [ ] レスポンスサイズが適切か
- [ ] キャッシュが適切に利用されているか
- [ ] タイムアウト設定が適切か

#### 6.3 フロントエンド
- [ ] 不要な再レンダリングがないか（React.memo等）
- [ ] 画像が最適化されているか
- [ ] バンドルサイズが適切か
- [ ] コード分割が適切か（動的import）

#### 6.4 その他パフォーマンス
- [ ] メモリリークがないか
- [ ] 無限ループ・再帰の制御があるか
- [ ] 重い処理が非同期化されているか

### 7. J-SOX準拠チェック

#### 7.1 監査証跡
- [ ] `docs/jsox-compliance.md` の監査ログ設計に準拠しているか
- [ ] すべての重要な操作がログに記録されているか
  - Who（誰が）: ユーザーID
  - When（いつ）: タイムスタンプ
  - What（何を）: 操作内容
  - Where（どこで）: IPアドレス、エンドポイント
  - Result（結果）: 成功/失敗
- [ ] ログが改ざん不可能な形で保存されているか

#### 7.2 職務分掌
- [ ] 権限が適切に分離されているか
- [ ] 承認フローが実装されているか（該当する場合）
- [ ] 兼務禁止の制御があるか（該当する場合）

#### 7.3 アクセス制御
- [ ] ロールベースアクセス制御（RBAC）が実装されているか
- [ ] 最小権限の原則が守られているか
- [ ] データアクセス権限が適切に制御されているか

#### 7.4 データ保護
- [ ] 個人情報・機密情報が適切に保護されているか
- [ ] データバックアップの仕組みがあるか
- [ ] データ削除時の監査ログが残るか

### 8. コード品質チェック

#### 8.1 静的解析
- [ ] リントエラーがないか（ESLint, golangci-lint等）
- [ ] 型チェックエラーがないか（TypeScript, Go）
- [ ] フォーマットが統一されているか（Prettier, gofmt）

#### 8.2 テスト
- [ ] 単体テストが書かれているか
- [ ] テストカバレッジが十分か（目安: 80%以上）
- [ ] E2Eテストが書かれているか（該当する場合）
- [ ] テストが成功しているか

#### 8.3 コードの可読性
- [ ] 適切な命名がされているか
- [ ] 複雑な処理にコメントがあるか
- [ ] 関数・メソッドが適切なサイズか（長すぎない）
- [ ] 重複コードがないか（DRY原則）

### 9. ドキュメント整合性チェック

#### 9.1 永続的ドキュメント
- [ ] `docs/glossary.md` が更新されているか（新規用語追加時）
- [ ] `docs/service-contracts.md` が更新されているか（API変更時）
- [ ] サービス別の `docs/` が更新されているか（アーキテクチャ変更時）

#### 9.2 ステアリングファイル
- [ ] `tasklist.md` のすべてのタスクが completed になっているか
- [ ] 実装中に判明した変更点が記録されているか

---

## レビュー実行フロー

1. **ステアリングファイル読み込み**: `.steering/$1/` 配下の全ファイルを読み込む
2. **対象サービス特定**: requirements.md から影響範囲を確認
3. **実装コード読み込み**: 対象サービスのコードを読み込む
4. **チェックリスト実行**: 上記9カテゴリのレビュー項目を順次チェック
5. **問題点のレポート**:
   - **Critical（重大）**: セキュリティ脆弱性、J-SOX違反、仕様の重大な逸脱
   - **High（高）**: 機能不足、パフォーマンス問題、API契約違反
   - **Medium（中）**: コード品質問題、用語不統一、ドキュメント未更新
   - **Low（軽微）**: コードスタイル、コメント不足
6. **改善提案**: 問題点の修正方法を具体的に提案

...

### Medium (中) - N件

---

## 注意事項

- レビューは自動化できる部分と人間の判断が必要な部分があります
- Critical/High問題は必ず修正してください
- Medium/Low問題は優先度に応じて対応してください
- レビュー結果を `.steering/$1/review-report.md` として保存します
