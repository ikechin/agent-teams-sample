# E2Eテストシナリオ

## 概要

本ドキュメントは、加盟店契約管理システムのE2Eテストシナリオを定義します。

**重要:** 新しい画面や機能を実装した際は、必ずこのドキュメントにテストシナリオを追加してください。

**対象読者:**
- E2E Test Agent（Agent Teams実装時）
- QA担当者
- 開発者全員

---

## テストシナリオの優先度

| 優先度 | 定義 | テスト実施 |
|--------|------|------------|
| **High** | ビジネス上重要なクリティカルパス、金銭処理、J-SOX要件 | 必須（リリース前に必ず実施） |
| **Medium** | 頻繁に使用される機能 | 推奨（時間があれば実施） |
| **Low** | 使用頻度が低い機能 | オプション（時間があれば実施） |

---

## テストシナリオ一覧

### 1. 認証・認可（Authentication & Authorization）

#### 1.1 ログインフロー
**優先度:** High
**実装状況:** ✅ 実装済み（`tests/auth/login-flow.spec.ts`）

**前提条件:**
- テストユーザーが存在する（`test@example.com` / `password123`）
- Frontend（Next.js）がポート3000で起動している
- BFF（Go Echo）がポート8080で起動している

**テストケース:**
1. 正常系: ログイン成功 → ダッシュボードにリダイレクト
2. 異常系: パスワード誤り → エラーメッセージ表示
3. 異常系: 未認証で/dashboardアクセス → /loginにリダイレクト

**テストデータ:**
```json
{
  "validUser": {
    "email": "test@example.com",
    "password": "password123"
  },
  "invalidPassword": {
    "email": "test@example.com",
    "password": "wrongpassword"
  }
}
```

---

#### 1.2 セッションタイムアウト
**優先度:** Medium
**実装状況:** ❌ 未実装

**前提条件:**
- ログイン済みユーザー

**テストケース:**
1. 30分間操作しないとセッションがタイムアウトする
2. タイムアウト後、自動的にログイン画面にリダイレクトされる

---

#### 1.3 権限チェック
**優先度:** High
**実装状況:** ❌ 未実装

**前提条件:**
- 複数の権限を持つユーザーが存在する

**テストケース:**
1. `contracts:read` 権限がないユーザーは契約一覧画面にアクセスできない
2. `contracts:approve` 権限がないユーザーは承認ボタンが表示されない
3. `roles:manage` 権限があるユーザーのみロール管理画面にアクセスできる

### 2. 加盟店管理（Merchant Management）

#### 2.1 加盟店一覧表示
**優先度:** High
**実装状況:** ✅ 実装済み（`tests/merchants/merchant-list.spec.ts`）

**前提条件:**
- ログイン済みユーザー（`merchants:read` 権限、contract-managerロール）
- BFFが加盟店APIモックデータを返却する状態

**テストケース:**
1. ログイン後に加盟店一覧画面にアクセスできること
2. モックデータ（2件: テスト加盟店1、テスト加盟店2）が表示されること
3. テーブルの各カラム（加盟店コード、名前、住所、担当者）が表示されること

**テストデータ（BFFモック）:**
```json
{
  "merchants": [
    {
      "merchant_code": "M-00001",
      "name": "テスト加盟店1",
      "address": "東京都渋谷区渋谷1-1-1",
      "contact_person": "山田太郎"
    },
    {
      "merchant_code": "M-00002",
      "name": "テスト加盟店2",
      "address": "東京都新宿区新宿2-2-2",
      "contact_person": "佐藤花子"
    }
  ]
}
```

---

### 3. 契約管理（Contract Management）

#### 3.1 契約CRUD操作
**優先度:** High
**実装状況:** ❌ 未実装

**前提条件:**
- ログイン済みユーザー（`contracts:create`, `contracts:update`, `contracts:delete` 権限）
- 加盟店が登録されている
- サービスが登録されている

**テストケース:**
1. 契約を新規登録できる
2. 契約詳細を表示できる
3. 契約を編集できる（金額以外）
4. 契約ステータスを変更できる
5. 必須項目が未入力の場合、バリデーションエラーが表示される

**テストデータ:**
```json
{
  "contract": {
    "merchant_id": "uuid-of-merchant",
    "service_id": "uuid-of-service",
    "start_date": "2026-05-01",
    "end_date": null,
    "initial_fee": 50000,
    "monthly_fee": 10000,
    "notes": "E2Eテスト用契約"
  }
}
```

---

#### 3.2 契約金額変更と承認フロー（J-SOX対応）
**優先度:** High
**実装状況:** ❌ 未実装

**前提条件:**
- ログイン済みユーザーA（`contracts:update` 権限）
- ログイン済みユーザーB（`contracts:approve` 権限）
- 契約が登録されている

**テストケース:**
1. ユーザーAが契約の月額費用を変更すると、承認申請が作成される
2. ユーザーBが承認一覧画面で申請を確認できる
3. ユーザーBが承認すると、契約が更新される
4. ユーザーBが却下すると、契約は変更されない
5. **職務分掌チェック**: ユーザーAが自分の申請を承認しようとするとエラーになる

**重要:** J-SOX要件（職務分掌）の確認

---

#### 3.3 契約変更履歴表示
**優先度:** High
**実装状況:** ❌ 未実装

**前提条件:**
- ログイン済みユーザー
- 契約が登録され、複数回変更されている

**テストケース:**
1. 契約詳細画面で「変更履歴」ボタンをクリックできる
2. 変更履歴一覧が時系列で表示される
3. 各変更履歴に「変更者」「変更日時」「変更内容」が表示される
4. 金額変更の場合、承認情報が表示される

**重要:** J-SOX要件（監査証跡）の確認

---

#### 3.4 契約検索・フィルタリング
**優先度:** Medium
**実装状況:** ❌ 未実装

**前提条件:**
- ログイン済みユーザー
- 複数の契約が登録されている

**テストケース:**
1. 契約番号で検索できる
2. 加盟店でフィルタリングできる
3. サービスでフィルタリングできる
4. ステータスでフィルタリングできる
5. 複数条件を組み合わせてフィルタリングできる

## クリティカルパス（必須テスト）

以下のシナリオは、リリース前に必ず実施する必要があります。

### パス1: 新規加盟店契約フロー（最重要）
1. ログイン
2. 加盟店を新規登録
3. 新規契約を登録
4. 契約詳細を確認

### パス2: 契約金額変更・承認フロー（J-SOX対応）
1. ユーザーA: ログイン
2. ユーザーA: 契約の月額費用を変更
3. ユーザーA: 承認申請を作成
4. ユーザーA: ログアウト
5. ユーザーB: ログイン（承認者）
6. ユーザーB: 承認一覧で申請を確認
7. ユーザーB: 承認
8. ユーザーA: ログイン
9. ユーザーA: 契約詳細で変更が反映されていることを確認
10. ユーザーA: 変更履歴を確認

### パス3: 新サービス展開フロー
1. ログイン
2. 新サービスを登録
3. 既存加盟店に新サービスの契約を追加
4. 契約一覧で新サービスの契約を確認
